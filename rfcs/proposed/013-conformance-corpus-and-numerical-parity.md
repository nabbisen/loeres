# RFC 013 — Conformance Corpus and Numerical Parity Policy

**Status.** Proposed
**Tracks.** Cross-cutting numerical evidence, cluster/device compatibility, regression testing, and release validation
**Touches.** `conformance/`, `xtask/conformance`, `loeres-device` examples, `loeres-cluster` examples, solver-specific test fixtures, CI workflows

---

### Extended Metadata
* **Rust Edition Compliance:** Rust 2024 Baseline
* **Target Environment:** Shared verification layer spanning `loeres-device`, `loeres-cluster`, `loeres-backend-static`, and `loeres-backend-std`

## 1. Executive Summary & Problem Statement

Loeres intentionally supports two different execution families: static device solving and dynamic cluster solving. These paths will not produce bitwise-identical outputs in general. They may differ because of BLAS implementations, sparse storage ordering, scalar precision, target floating-point behavior, and solver configuration.

The project still needs evidence that both families implement the same mathematical contract. This RFC defines the conformance corpus and numerical parity policy used to validate that equivalent problem instances produce compatible outcomes within declared tolerances.

## 2. Architectural Context & Dependency Alignment

The conformance corpus is a verification artifact, not a runtime dependency. Device crates do not parse large corpus files or pull in serialization frameworks.

Dependency alignment:

| Component | Relationship to this RFC | Dependency impact |
|---|---|---|
| `conformance/` | Stores fixtures and expected outcomes | Repository test data only |
| `xtask conformance` | Loads fixtures and runs comparisons | Host-only; may use `std` |
| `loeres-device` | Exposes testable deterministic solver entrypoints | No parser dependency required |
| `loeres-cluster` | Exposes dynamic solver entrypoints | `std` allowed |
| `loeres-core` | Owns shared status/error categories | No `std`, no `alloc` |

## 3. Concrete Technical Specification

### 3.1 Corpus directory layout

The repository must define a stable corpus layout:

```text
conformance/
  README.md
  smoke/
    dense-qp-small-001.toml
    projected-gradient-small-001.toml
  extended/
    dense-qp-conditioned-001.toml
    sparse-qp-medium-001.toml
  adversarial/
    invalid-dimensions-001.toml
    non-finite-input-001.toml
    singular-system-001.toml
  expected/
    smoke.toml
    extended.toml
    adversarial.toml
```

The exact fixture format is deferred to implementation, but it must be host-side only. Device examples should receive already-materialized static arrays generated by tests or fixtures.

### 3.2 Fixture metadata

Each fixture must declare:

* fixture ID;
* problem class;
* dimensions;
* scalar family;
* validation state requirement;
* solver family;
* configuration profile;
* expected outcome category;
* tolerance policy;
* deterministic seed, if generated;
* whether the fixture is smoke, extended, adversarial, or regression-only.

### 3.3 Outcome comparison model

The default parity rule is tolerance-based convergence, not bitwise equality.

The baseline cross-path tolerance is:

```text
epsilon = 1e-5
```

Solver-specific RFCs may define tighter or looser tolerances, but they must justify the value and identify whether it applies to objective value, primal residual, dual residual, variable vector distance, or status classification.

### 3.4 Required comparison categories

`cargo xtask conformance` must report:

| Category | Meaning |
|---|---|
| `status_match` | Device and cluster return compatible `loeres_core::solver::SolveStatus` values (RFC 014); non-convergence is `Ok(NotConverged)`, never an error |
| `objective_within_tolerance` | Objective values differ by no more than declared tolerance |
| `residual_within_tolerance` | Constraint/residual measures satisfy declared tolerance |
| `solution_within_tolerance` | Solution vectors are compatible under the selected norm |
| `expected_failure_match` | Both paths reject invalid/adversarial input with compatible error categories |
| `not_comparable` | Fixture is intentionally supported by one path only |

A single scalar “pass/fail” output is insufficient for review. The command must expose the category that failed.

### 3.5 Smoke, extended, and adversarial suites

The smoke suite must run quickly and cover the minimal public contract.

The extended suite may include larger matrices, sparse structures, target-specific comparisons, and conditioning stress cases.

The adversarial suite must cover at least:

* invalid dimensions;
* non-finite scalar input for float scalars;
* singular or ill-conditioned structures;
* invalid tolerance configuration;
* max-iteration exhaustion;
* workspace/problem mismatch for device cases.

Fixtures must be stratified by conditioning so the bounded-iteration promise is tested honestly rather than assumed. For strongly convex smooth problems the iteration count generally worsens with conditioning, so each fixture must record its `(condition estimate, iteration cap, epsilon)` triple, and "Converged" is only an expected outcome when the cap is set relative to conditioning. The rate intuition (poorer conditioning needs more iterations) is guidance, not an exact universal bound for all projected-gradient cases.

| Fixture class | Expected v0.x outcome |
|---|---|
| Strongly convex, well-conditioned box QP (cap set relative to conditioning) | `Ok(Converged)` within cap and epsilon |
| Strongly convex, ill-conditioned box QP | `Ok(NotConverged)`; bounded and finite |
| PSD / merely-convex box QP | `Ok(NotConverged)`; bounded and finite |
| General linear-inequality QP (out of v0.x kernel scope) | `Err(UnsupportedProblemStructure)`, not `InvalidInput` |
| Non-finite input | `Err(NonFiniteInput)` |
| Numerically invalid operation | `Err(NumericalDomain)` or `Err(IllConditioned)` by cause |

### 3.6 Corpus generation policy

Generated fixtures must record their seed and generator version. Regenerating a fixture without changing its ID is forbidden unless the generated values are intentionally identical.

If a fixture changes semantically, it receives a new ID. Historical fixtures may be moved to regression-only status but should not be silently deleted.

### 3.7 Problem-family staging

The first conformance corpus should focus on the first accepted device solver. Until RFC 006 chooses the baseline kernel, the corpus may contain placeholder categories but must not over-promise support for interior-point, mixed-integer, or nonlinear models.

## 4. Rust Systems-Level Nuances & Memory Safety

The corpus runner may allocate on the host. Device crates may not depend on fixture parsers.

For device tests, host-side tooling may generate Rust test modules or static arrays from fixtures. Generated files must be reviewable and deterministic.

Large fixture data should not be embedded into device firmware by default. Device conformance builds should use tiny smoke fixtures unless an explicit extended profile is selected.

Comparing floating-point vectors must avoid naive absolute-only checks where scale matters. The comparison policy should support absolute tolerance, relative tolerance, and norm-based comparison.

## 5. Algorithmic & Numerical Fail-Safe Guardrails

The corpus must validate failure behavior as well as success behavior. A solver that panics, hangs, allocates unexpectedly, or returns an unstructured error fails conformance even if normal cases converge.

Minimum fail-safe expectations:

1. invalid fixtures return structured error categories;
2. ill-conditioned fixtures do not cause unbounded iteration;
3. max-iteration fixtures terminate with `Ok(SolveReport)` carrying `SolveStatus::NotConverged` and `TerminationReason::IterationCap` (RFC 014), not a `SolverError`;
4. trusted-input fixtures still surface numerical-domain errors discovered during computation;
5. unsupported problem classes return unsupported-problem rather than attempting best-effort execution.

## 6. Verification, Validation, and CI Gates

Acceptance gates:

1. The repository contains `conformance/README.md` describing fixture format and status categories.
2. `cargo xtask conformance --suite smoke` runs in standard CI.
3. Extended and adversarial suites are available for release CI.
4. The runner reports per-fixture and per-category results.
5. At least one fixture checks expected failure behavior.
6. Cross-path numerical comparison uses tolerance-based rules, never bitwise identity as the baseline.
7. Solver-specific RFCs must add or update fixtures when they introduce new supported problem classes.
